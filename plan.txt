1. Docker (The "It Works on My Machine" Solver)
The Problem: Installing a raw PostgreSQL database directly onto a Windows machine is notoriously difficult. 
If you mess up a configuration file, it embeds itself deep into your Windows registry and is a nightmare to uninstall. 
Furthermore, adding the pgvector machine learning extension to a Windows Postgres installation often requires compiling raw C++ code by hand.
The Solution (Why we use it): Docker creates a tiny, invisible, fake computer (a "container") inside your laptop. 
The developers of pgvector already did the hard work of installing Postgres and the ML extension perfectly inside this fake computer. 
By running Docker, you just download their perfect setup. If you ever break it, you don't ruin your Windows 
machine—you just delete the container and download a fresh one in two seconds.

2. Ngrok (The Cellular Bridge)
The Problem: Your Python FastAPI server is running on localhost:8000. "Localhost" literally means "this specific computer." 
If you pick up your physical iPhone or Android and try to go to localhost:8000, 
the phone will look for a server running inside the phone itself, and it will fail.
The Solution (Why we use it): Ngrok acts as a secure wormhole. It generates a public, secure internet link 
(like https://random.ngrok.app) and tunnels it directly through your Wi-Fi router straight into your laptop's port 8000. 
This allows you to stand in the middle of a grocery store on 5G cellular data, take a picture of food, 
and have it successfully hit the Python server sitting on your desk at home.

3. WatermelonDB (The Offline Savior)
The Problem: Most React Native apps use standard local storage to save data, which loads the entire database into the phone's RAM 
all at once. If your user saves 500 recipes and 200 pantry ingredients, the app will freeze and stutter every time they open it.
The Solution (Why we use it): WatermelonDB is "lazy." It uses an underlying SQLite database on a separate thread. 
If the user has 500 recipes, WatermelonDB only loads the 10 they are actively looking at. Crucially, 
it allows your app to work completely offline—so if the user is deep inside a concrete grocery store with no cell service, 
they can still check their virtual pantry without the app crashing.

4. TanStack Query (The Network Manager)
The Problem: If you use basic JavaScript to ask the server for a recipe, and the Wi-Fi drops for one second, the request dies.
The app throws an ugly red error, and the user has to manually hit "refresh."
The Solution (Why we use it): TanStack handles network chaos automatically. 
If a request fails, TanStack quietly retries it 3 times in the background before bothering the user. 
It also caches (remembers) the data. If the user clicks a recipe, hits the back button, and clicks the recipe again, 
TanStack instantly shows the cached version instead of forcing the app to re-download it from Python.

5. SQLAlchemy (The Database Translator)
The Problem: To get data out of a database, you have to write raw SQL strings (like SELECT * FROM recipes WHERE id = 5). 
Doing this inside Python is incredibly tedious, prone to typos, and opens your app up to "SQL Injection" hacks.
The Solution (Why we use it): SQLAlchemy is an ORM (Object-Relational Mapper). 
It lets you use normal Python code (like db.query(Recipe).filter(id=5)) to talk to the database. 
It automatically translates your Python into perfect, hacker-proof SQL behind the scenes.

6. scikit-learn & pgvector & Sentence-Transformers(The Math Brains)
The Problem: Standard databases can only search for exact text matches. If a user searches for "beef," 
a standard SQL database will not return a recipe for "steak" because the words are spelled differently.
The Solution (Why we use it): We use sentence transformers in Python to convert the word "beef" into a 
vector array of mathematical coordinates (e.g., [0.5, 0.9, -0.2]). 
We use pgvector in PostgreSQL because a normal database cannot store or compare these arrays. 
pgvector allows the database to do geometry, finding the "closest" recipes in mathematical space, 
meaning "steak" and "beef" will be recognized as matches.

7. Expo EAS (The Cloud Compiler)
The Problem: To turn your React Native code into a real iOS app, Apple forces you to own a Mac computer and use Xcode.
The Solution (Why we use it): Expo Application Services (EAS) has giant server farms of Mac computers. 
When you finish your app, you send your code to EAS, and their Macs compile the final .ipa or .apk files for you. 
It completely bypasses the hardware restrictions of mobile development.

8. What is Axios doing?
Axios is simply the mailman for your frontend.
Your React Native app holds JavaScript objects (e.g., ["Tomato", "Onion"]).
The Python server only understands JSON text over the HTTP protocol.
Axios's Job: It takes your JavaScript array, automatically converts it into a JSON string, attaches the correct headers, 
securely sends the HTTP POST request to your Ngrok URL, waits for Python to reply, 
and translates the returned JSON back into a JavaScript object so React can display it.

9. What is _layout.tsx and "Global Context"?
In Expo Router, _layout.tsx is not just a visual wrapper (like a header or footer). 
It is the absolute "root" of your application.
What is Global Context? Imagine a WatermelonDB database instance or a TanStack Query cache. 
If you put them in one specific screen, the other screens can't see them. 
"Context" (specifically the React Context API) is an invisible bubble that wraps your entire app.
Why in _layout.tsx? By initializing WatermelonDB and TanStack inside _layout.tsx, 
you wrap your entire app in their "Providers." This means every single screen in your app can instantly 
reach up and grab data from the offline pantry without you having to manually pass the database connection down screen by screen.
